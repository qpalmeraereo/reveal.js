What is a Commit?

A commit is a

  • Hash
  • Diff
  • Author/message/etc.
  • Parents (more detail to come)

*Under the hood there is more happening, but most people (including you) can treat it as being this simple

How is a Commit Made?

A (non-merge) commit is built by:

  • Compressing the diff
  • Setting the meta information (author/time/message)
  • Setting the parent as the current HEAD

What is HEAD?

A symbolic reference to the tip of the current branch

What is a Branch?

We skipped over one part of how a commit is built. The current branch's pointer is moved forward when you build a new commit. If you remember this slide this implicitly moves HEAD forward as well.

Making a New Branch

This creates a new symbolic reference to the current HEAD with the name that you provide

  • DOES NOT change HEAD
  • DOES NOT make any new commits
  • DOES NOT change any code

Switching Branches

This moves HEAD to the tip of the specified branch

This DOES change the files to what the branch knows about them

Making Commits on Other Branches

HEAD is still on the "dev" branch, so it moves the dev branch forward (leaving master where it was)

Move code from one branch to another

You likely want whatever changes you've put on the branch to be merged in with other code

One way to do this is

git merge

There are two kinds of merging

  • Fast-Forwarding
  • Non Fast-Forwarding

Fast-Forward Merge

If the branch that you're merging into is a direct ancestor with no children then you can fast-forward

This is really cheap

Because branches are effectively named pointers to commits it is trivial to point them somewhere else

Non Fast-Forward Merge

If you want there to be history of your branch or if the branch you're merging into has changed since you branched off

This makes a new commit "N"

This is also really cheap

All that was made was a new commit object with two parents

Quick: What's a Cherry Pick?

Cherry picking is copying the information from one commit and placing it somewhere else with a new hash

This makes a new commit which has the same content as I, but a different hash

Merging with Cherry Picks

Given what we know about cherry picking, we could logically copy branches with a series of cherry picks

Cherry Picking isn't that fun

Looking at the previous cherry pick example, it is inconvient and does not scale well to move lots of commits with cherry picks

It would be nice if you could do batches of them together, right? Git has provided this for us, it's called...

What is rebasing then?

Rebasing just moves a series of commits

Under the hood rebasing is just a shell script that uses cherry picking

Common use cases of rebasing

Git pull --rebase

This pulls from the remote (raise your hand if you don't know what this is) and then applies your local commits on top of the new HEAD

Git rebase -i

This is the same rebase that we've discussed, but it allows you to pause between certain commits to make changes

git rebase --onto

This allows you to change the parent of a commit (useful to move where a branch branches)

Navigating in the DAG

The two primary commands that we'll talk about are:

git checkout

This moves HEAD around

*This does not move any branches

**This leaves you in a detached head mode (you cannot commit)

git checkout (branches)

If you checkout a branch, you move HEAD to that branch

*This still does not move master

**You are now on dev, and can make commits to dev

git reset

This moves branches around

*M still exists, but is unreachable

**Any uncommited changes to the filesystem are lost

Checkout vs. Reset

What is the formal difference?

  • Checkout moves HEAD (and not branches)
  • Reset moves branches (and HEAD comes with)
